01 — Visão

O que é o Squad Autônomo

O Squad Autônomo é uma camada de orquestração que coloca agentes Claude em pair-programming com pessoas, organizados num quadro Kanban. Cada funcionalidade percorre uma esteira (Discovery → Planejamento → Desenvolvimento → Code Review → QA → Concluído), e cada transição de etapa é um portão humano: nada avança sem alguém aprovar. O GitHub é a fonte da verdade — todo artefato e todo código vivem em branches reais do repositório.

A diferença para um "agente autônomo" tradicional é filosófica: aqui o objetivo não é tirar a pessoa do circuito, é amplificar o time. O humano decide o que entra, revisa o que sai, e mantém o controle de custo e qualidade. O agente faz o trabalho pesado e repetitivo com rastreabilidade total.

Modelo mental

Kanban + gates

Cada card é uma feature. Cada raia é uma etapa. Entre raias, um portão humano.

Fonte da verdade

GitHub

Branches por ambiente, PRs de elevação, artefatos versionados. Auditável.

Execução

Claude Managed Agents

Agentes especializados por papel rodam em infra gerenciada da Anthropic.

02 — Arquitetura

Componentes envolvidos

Quatro blocos cooperam. O App Squad (Next.js na Vercel) é o cérebro de orquestração e a interface; o Supabase guarda estado, autenticação, métricas e gates; o GitHub é onde o código nasce e vive; e os Claude Managed Agents são os trabalhadores que executam cada etapa.

cérebro

App Squad

Orquestra a esteira, monta os prompts de cada etapa, aplica gates e calcula custo. Inclui o Squad Assistant.

memória

Supabase

Auth, Postgres (cards, gates, métricas, pessoas, ambientes), Realtime para o board e Storage para protótipos.

músculo

Claude Agents

Sessões gerenciadas por papel. Recebem o contexto, clonam o repo e trabalham na branch alvo.

verdade

GitHub

Branches por ambiente, PRs de elevação, todo artefato versionado e auditável.

Segredos

As chaves (Anthropic, GitHub, Supabase service role) vivem apenas nas variáveis de ambiente da Vercel — nunca no código. O repositório só contém .env.example com placeholders.

Setup de agentes

A primeira vez, em /admin/setup, semeia e implanta os 7 agentes builtin (PM, Tech Lead, 3 Devs, Code Reviewer, QA). Cada novo time criado herda esse conjunto automaticamente.

04 — Novo time

Como configurar um Time

Em Settings, a tela abre como uma lista de times. Clique em configurar num time (ou + novo time) para abrir todas as configurações dele — cada time é independente e o seletor do board alterna o contexto.

  1. Aplicações. Cadastre um ou mais repositórios. Cada aplicação carrega: repo GitHub, branch base, tipo (nova ou existente/legado), natureza, stack de tecnologias, arquivo de instruções (CLAUDE.md) e dependências entre aplicações.
  2. Ambientes (branches). Dentro de cada aplicação, defina os ambientes — cada um é um nome + uma branch (ex.: Dev → develop, Homologação → homolog, Produção → main) — e a cadeia de promoção (Dev promove para Homologação, que promove para Produção).
  3. Pessoas & custos. Cadastre as pessoas por tipo — PM, Tech Lead, Dev, QA (espelhando os agentes) — com salário e horas/mês. É daqui que sai o custo humano de cada etapa.
  4. Conhecimento & Dreaming. Aponte fontes de conhecimento (docs, links, padrões) que os agentes leem antes de trabalhar. O Dreaming faz o onboarding do time gerando instruções de contexto.
  5. Orquestração & agentes. Personalize prompt e modelo de cada agente, se quiser fugir do padrão.
  6. Workflow, gates & notificações. Auto-merge, branch base padrão, revisão reforçada (exige digitar APROVAR em caminhos sensíveis), caminhos sensíveis, e o webhook do Microsoft Teams que avisa o time a cada etapa concluída.

06 — Nova feature

Como criar uma nova funcionalidade

  1. No board, clique em + nova feature.
  2. Informe título e descrição (o briefing que vira o PRD).
  3. Escolha a aplicação (qual repo) e o ambiente (qual branch o trabalho usa).
  4. Opcional: branch nova — em vez de usar a branch do ambiente, defina um nome novo e a branch raiz de onde clonar. Se a branch não existir, é criada a partir da raiz.
  5. Opcional: anexe protótipos HTML que o Dev Agent de frontend deve seguir 1:1.
  6. Confirme. O card entra na raia Discovery e a esteira começa, com gate em cada etapa.

No diálogo de aprovação de cada etapa, você vê o agente, o modelo, a branch alvo e a branch raiz antes de disparar — e pode editar o briefing inline antes de aprovar.

07 — Métricas & custo

Quais métricas você tem

Cada card mostra quatro indicadores, todos calculados automaticamente:

tempo

Cycle Time

Da criação ao "concluído". Em horas, ao vivo.

qualidade

Aprovação

Passou de primeira? Taxa de aprovação na 1ª tentativa (dos gates).

testes

Cobertura

Vem do qa-report do QA Agent. Meta ≥ 80%.

dinheiro

Custo

Total em R$ = horas humanas + tokens. Detalhado por etapa.

Como o custo é calculado — por etapa, somando até a feature

O grande diferencial é o custo por etapa, incremental. Cada execução grava quanto consumiu, e a feature acumula passo a passo:

  • Custo humano da etapa = duração real da etapa × custo/hora do papel correspondente. O papel vem do agente: Discovery usa o custo médio de uma pessoa do tipo PM; Planejamento/Code Review, do Tech Lead; Desenvolvimento, do Dev; QA, do QA. Ex.: Discovery levou 1h e o PM médio custa R$ 90/h → R$ 90 de custo humano.
  • Custo de tokens da etapa = tokens consumidos × preço oficial do modelo (em USD), já convertido para reais pelo câmbio USD→BRL configurado no time.
  • Total da etapa = humano + tokens. A tabela "custo por etapa" lista cada execução (com o modelo usado) e o acumulado da feature.
Por que isso importa

Em vez de só "quanto tempo levou", você enxerga quanto custou de verdade entregar a funcionalidade — humano + IA — para decidir o que vale automatizar e onde o gasto se concentra. Os Dashboards agregam isso em tendências semanais.

08 — Código no GitHub

Como visualizar o código gerado

  • Artefatos no card. No detalhe do card, a aba mostra os artefatos por etapa (PRD, ADRs, código, relatório de QA), lidos diretamente da branch do ambiente daquele card.
  • Edição inline. Clique num artefato para abrir; o botão editar permite alterar e salvar direto no arquivo no GitHub (commit na branch do ambiente).
  • Bilíngue. O canônico é inglês; o toggle EN · ver PT traduz para leitura. Editar em PT e salvar grava em inglês automaticamente.
  • Elevação de ambiente. No card concluído, "elevar ambiente" abre um único PR da branch atual para a branch do ambiente destino (ex.: develop → homolog), seguindo a cadeia configurada.
  • Resumo de infraestrutura. Ao final, um resumo gerado com Opus consolida tudo que a feature precisa para rodar (compute, bancos, filas, secrets) — com um bloco JSON pronto para provisionamento via MCP.
09 — Por que AI-native

Por que essa solução num contexto AI-native

Uma empresa AI-native não trata IA como um plugin — ela redesenha o processo de engenharia em torno de agentes, mantendo governança humana. O Squad Autônomo materializa isso:

governança primeiro

Humano no controle

Gates obrigatórios em cada etapa, revisão reforçada em áreas sensíveis e trilha de auditoria nos gates. A IA acelera; a pessoa decide.

rastreabilidade

GitHub como verdade

Nada acontece "numa caixa-preta": todo artefato e todo commit estão em branches reais, revisáveis e reversíveis.

economia mensurável

Custo por etapa

Você sabe o custo humano + IA de cada entrega. Decisões de automação deixam de ser fé e passam a ser número.

especialização

Papéis de SDLC

PM, Tech Lead, Dev e QA como agentes distintos — espelhando o time real, não um único agente "faz-tudo".

multi-ambiente

Promoção hierárquica

Dev → Homologação → Produção com PRs de elevação, do jeito que engenharia madura já opera.

infra gerenciada

Sem operar o motor

Os agentes rodam em infra gerenciada da Anthropic; o time foca no produto, não em manter um runtime de agentes.

10 — Comparativo

Frente às demais soluções de orquestração

O mercado de 2026 tem três famílias: frameworks DIY (LangGraph, CrewAI, AutoGen, OpenAI Agents SDK) — bibliotecas para você montar o loop do agente; agentes autônomos de software (estilo Devin / coding agents) — um agente que tenta fazer tudo sozinho; e o Squad Autônomo — uma camada de produto e governança sobre agentes gerenciados. Um aprendizado consistente da literatura de 2026: os frameworks moldam o loop do agente, mas nenhum entrega, sozinho, a camada de governança de produção (aprovação humana, auditoria, rollback). É exatamente essa camada que o Squad já provê.

Pontos de destaque — por que continuar com o Squad

  • Você não precisa construir a camada de governança. Em LangGraph & cia. você teria que implementar gates, auditoria, custo e promoção do zero — semanas de engenharia. O Squad já entrega isso.
  • Controle sem abrir mão de velocidade. Diferente de um agente autônomo "caixa-preta", aqui cada passo é revisável e reversível — sem voltar à lentidão de fazer tudo à mão.
  • Transparência financeira real. O custo por etapa (humano + IA, em R$) é raro nas outras soluções e é o que permite decidir, com número, o que vale automatizar.
  • Aderência ao processo de engenharia que já existe. Kanban, papéis, ambientes, PRs — o time adota sem reaprender a trabalhar.
Sendo justo — quando outra opção faz sentido

Se o requisito for independência de provedor (trocar de LLM livremente) ou um loop de agente muito atípico, um framework como LangGraph dá mais controle de baixo nível — ao custo de você construir a governança. Se o objetivo for automação totalmente hands-off e o time aceita menos controle e auditoria, um agente autônomo puro entrega rápido. O Squad é a escolha quando você quer governança + especialização + custo claro com baixo esforço de manutenção — o caso de uma engenharia AI-native em ambiente regulado como o nosso.

11 — Papéis humanos

O que cada perfil faz, valida e como pede alteração

Cada perfil tem um momento-chave: o portão (gate) da sua etapa. É ali que a pessoa lê os artefatos, valida e decide se libera o avanço. O diálogo de aprovação é o mesmo padrão para todos — mostra o agente, o modelo, a branch alvo, deixa editar o briefing inline antes de disparar, e quando toca área sensível exige digitar APROVAR (revisão reforçada).

PM

Product Manager — gate de Discovery

o que faz

Escreve e refina o briefing, prioriza e define o porquê da funcionalidade.

o que validar no PRD

Escopo correto; critérios de aceite testáveis e sem ambiguidade; fluxos e personas certos; requisitos não-funcionais (segurança, performance); e se o infrastructure.md capturou o que será necessário.

como pedir alteração

Edita o briefing inline no diálogo antes de disparar; ou reprova o gate com um motivo — o motivo volta para o PM Agent refazer.

TL

Tech Lead — gate de Planejamento

o que faz

Garante coerência arquitetural e viabilidade técnica do que foi planejado.

o que validar nos ADRs / plano

Decisões arquiteturais sãs e justificadas; dependências entre aplicações respeitadas; riscos mapeados; quebra em chunks coesa e na ordem certa; e o portão de infraestrutura preenchido (nada criado sem aprovação).

como pedir alteração

Edita o plano inline; reprova com direcionamento técnico; ajusta a ordem dos chunks antes de liberar o desenvolvimento.

DEV

Dev — gate de Desenvolvimento & Code Review

o que faz

Valida o código de verdade — o que foi escrito, como, e se está na branch certa.

o que validar no código

Aderência ao plano; qualidade e legibilidade; testes presentes e significativos; nenhum segredo hardcoded; caminhos sensíveis tratados (a revisão reforçada sinaliza); commits na branch do ambiente/feature correta.

como pedir alteração

Edita o artefato inline e salva direto no GitHub; reprova com a lista de ajustes; usa o APROVAR da revisão reforçada como checkpoint consciente.

QA

QA — gate de QA

o que faz

Confirma que a entrega faz o que foi pedido, com qualidade comprovada.

o que validar no qa-report

Cobertura ≥ 80%; casos de borda cobertos; todos os critérios de aceite do PRD atendidos; suíte rodando verde (integrar os PRs dos chunks antes, se necessário).

como pedir alteração

Reprova com os cenários que faltam; devolve para o Dev refazer; só libera "concluído" quando a evidência fecha.

As 4 formas de pedir alteração

1) Editar o briefing inline no diálogo antes de disparar (refina o pedido sem reprovar). 2) Reprovar o gate com um motivo — o texto volta para o agente, que refaz considerando o feedback. 3) Editar o artefato e salvar no GitHub (correção pontual feita pela própria pessoa). 4) Revisão reforçada — digitar APROVAR em caminhos sensíveis, forçando uma conferência consciente.

PM

prd.md & acceptance-criteria.md & prototype.html

o que validar

PRD: o problema certo, escopo numerado sem furos, "fora de escopo" explícito, riscos reais. Critérios: cada requisito tem cenário positivo e negativo, redigidos de forma testável (Gherkin Given/When/Then). Protótipo: as telas batem com o que você espera — o PRD não pode descrever UI que o protótipo não mostra.

como pedir alteração

Edite o briefing inline no gate de Discovery antes de disparar; ou reprove com o motivo (ex.: "faltou cenário negativo para upload inválido"). O PM Agent reescreve o PRD/critérios. Pequenos ajustes de texto: abra o prd.md no detalhe do card, edite e salve no GitHub.

TL

adr.md & build-order.md & chunks

o que validar

ADR: cada decisão tem justificativa e alternativas consideradas; nada de escolha arbitrária. Chunks: cada critério de aceite do PRD está coberto por ≥1 chunk; o escopo de cada chunk é fechado; build-order respeita dependências (ex.: schema antes da API). Sub-issues com labels corretas.

como pedir alteração

Reprove o gate de Planejamento com direcionamento técnico (ex.: "trocar fila por outbox; quebrar o chunk 4 em dois"). Você também pode editar o adr.md/build-order.md inline e salvar. O Tech Lead Agent re-decompõe respeitando o feedback.

DEV

código, testes, infrastructure.md & onboarding

o que validar

Código: dentro do escopo do chunk, aderente ao ADR, testes presentes, sem segredo hardcoded, branch correta. infrastructure.md: toda necessidade está registrada e parada aguardando sua decisão (nada foi criado sozinho). Onboarding (base existente): CLAUDE.md descreve build/test/run reais.

como pedir alteração

Reprove o gate de Desenvolvimento com a lista de ajustes; para correções pontuais, edite o arquivo inline e salve no GitHub. No infrastructure.md, responda cada item com "criar novo" ou "usar existente X" — é assim que você libera (ou barra) a criação de recursos.

QA

qa-report.md & arquivos de teste

o que validar

coverage ≥ 80% na linha de topo; a tabela Scenarios Covered mapeia cada cenário do Gherkin para um teste com status pass; Implementation Bugs Found está preenchida (ou diz explicitamente que não há). Os bugs viraram comentários nos PRs/issues, não foram "consertados em silêncio".

como pedir alteração

Reprove o gate de QA listando os cenários que faltam ou a cobertura insuficiente. O QA Agent escreve os testes faltantes; se achou bug de implementação, ele devolve para o Dev (status:bug) em vez de mascarar.

infrastructure-summary.md — o fechamento

Ao concluir, o orquestrador gera com Opus um resumo de tudo que a feature precisa para rodar — compute, bancos (com caminho do script de migração e env var), filas, storage, serviços externos, secrets (só os nomes, nunca valores) e rede — terminando com um bloco json "machine_readable" pronto para o provisionador consumir. Valide se reflete o que foi realmente construído antes de levar para a esteira de provisionamento.

13 — Validação local

Validar a solução: branch → VS Code + Claude Code

Ler os artefatos no app comprova o processo; rodar a aplicação comprova o resultado. O fluxo recomendado é puxar a branch daquele card para o VS Code com a extensão do Claude Code, subir o ambiente local e validar que os desenvolvimentos batem com o PRD.

  1. Pegue a branch do card. Ela aparece no diálogo de gate (branch alvo) e no detalhe do card — é a branch do ambiente/feature onde os agentes commitaram.
  2. Traga para o VS Code. Primeira vez: git clone <repo>. Depois: git fetch origin && git checkout <branch-do-card>.
  3. Instale a extensão Claude Code. No VS Code (1.98+), abra Extensions (Ctrl/Cmd+Shift+X), busque Claude Code (publisher Anthropic) e instale — ou pelo Marketplace. Abra pelo ícone Spark na barra lateral.
  4. Suba a aplicação. Siga o CLAUDE.md que os agentes mantiveram (comandos de build/test/run) e o infrastructure.md (o que precisa existir — banco, fila, secrets).
  5. Use o Claude Code para acelerar. Peça em linguagem natural: "rode os testes", "suba o ambiente local", "explique o que mudou nesta branch em relação à main", "valide os critérios de aceite do PRD". A extensão lê o repo, roda comandos no terminal integrado e mostra os diffs com aceitar/rejeitar.
  6. Decida. Bateu com o PRD e rodou verde → eleve o ambiente (PR de promoção no app). Faltou algo → reprove o gate com o que ajustar, e o squad refaz.
Bom saber

A extensão embute a CLI: o que o painel gráfico não expõe, você roda no terminal integrado (claude). Servidores MCP são configurados pela CLI e a extensão os reaproveita — útil para conectar o mesmo provisionamento de infraestrutura que o resumo final da feature descreve.

Fonte da verdade, sempre

Validou e corrigiu algo localmente? Commite na branch do card e deixe o app registrar — assim o histórico, os artefatos e o custo continuam refletindo a realidade. Evite "consertos" que só existem na sua máquina.
